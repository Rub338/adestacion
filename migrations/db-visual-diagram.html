🔗 Relaciones entre tablas

auth.users ↔ perfiles

1:1 — Cada usuario de autenticación tiene un perfil. El admin lee nombre y rol de aquí.

salas → equipos

1:N — Una sala tiene muchas máquinas. FK: equipos.sala_id

equipos → registros

1:N — Una máquina tiene muchos registros (incidencias/mantenimiento). FK: registros.maquina_id

perfiles → registros

1:N — Un usuario puede crear muchos registros. FK: registros.usuario_id

registros → seguimientos

1:N — Una incidencia tiene muchas notas de seguimiento. FK: seguimientos.incidencia_id

perfiles → seguimientos

1:N — Un usuario puede escribir muchas notas. FK: seguimientos.usuario_id

📋 Resumen de cambios

🔴 Se eliminan

• usuarios_backup — tabla residual
• equipos: ancho_mm, alto_mm, profundidad_mm, frecuencia_dias, ultimo_mantenimiento
• registros: maquina_nombre, sala_nombre, operario_nombre, operario_email
• seguimientos: usuario_nombre
• Datos: todo el historial de registros y seguimientos

🟢 Se añaden / mejoran

• FK real en registros.usuario_id → perfiles.id
• NOT NULL en registros.maquina_id y seguimientos.incidencia_id
• RLS policy en perfiles: todos los auth pueden leer nombre+rol
• Nombres se resuelven por JOIN en tiempo real

⚠️ Impacto

• Salas, equipos y perfiles: se preservan
• Registros y seguimientos: se borran (historial desde cero)
• Login superadmin hardcodeado: se elimina
• Cambiar nombre de máquina/usuario: se refleja automáticamente en todo el historial
